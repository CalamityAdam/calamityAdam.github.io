Verdict: moving the coach cache TTL from 5 minutes to 1 hour saves $911.72 over 6 days (30.7% reduction), ≈ $55k/yr at current volume. Cache writes drop 15,476 → 2,014 (87% fewer write events). Per-request amortization improves 4.6× → 41.8×.
Hypothesis check (trusted_education / trusted_connect): the original hunch was that the long-tail Trusted intents, with their 8K+ token prompts, would benefit most from longer caching. The data refutes this. Trusted intents fire too rarely to fill a 1-hour window — at 1h, most still trigger a fresh write per session burst, but now at the 2× write premium instead of 1.25×. The biggest savings come from high-volume budgeting / debt / savings / housing / investing intents, where 1h captures longer reuse windows. Several individual trusted_connect.* intents actually cost more at 1h (e.g. trusted_connect.mortgage.home_purchase +44%, trusted_connect.insurance.long_term_care +55%). See the "Biggest losers" table below.
Better-than-uniform-1h: a per-intent TTL policy — 1h for high-volume groups, leave low-volume trusted_* on 5m — would push savings further. Worth a follow-up if the engineering cost is small.

Methodology & caveats

  • Window: 2026-04-25 → 2026-04-30 (6 full CT days, post 100%-Sonnet rollout on 2026-04-24). May 1+ excluded — Redshift ETL had only partially loaded May 1 and not loaded May 2/3 at extract time. 6 days short of the requested 7, but the alternative (including the Apr 24 ramp day) would mix pre-rollout traffic.
  • Filter: step_costs where agent_type='conversational-coach' and provider='anthropic', joined to turn_traces.active_intent_key. 86,130/86,130 steps had an attributable intent.
  • Pricing (Sonnet 4.6, per M tokens): input $3.00 · 5m write $3.75 (1.25×) · 1h write $6.00 (2×) · cache read $0.30 (0.1×) · output $15.00.
  • Simulator: for each coach step, gap to prior same-intent step → hit if ≤ TTL else write. Uses observed per-step cache_read_tokens + cache_write_tokens as the cacheable prefix size, so the simulator is anchored to real per-request bytes rather than estimates.
  • Validation: simulated 5m total = $2,968.22 vs observed $2,996.61, ratio 0.991. Tight enough to trust the counterfactual delta directionally and to within a few percent in magnitude.
  • Simplification: the simulator collapses the two real cache breakpoints (base prompt ~13.5K tokens + intent prompt ~3-11K tokens) into one prefix per intent. In reality the base portion almost always hits cache regardless of regime, so the savings analysis isolates correctly to the intent-block churn — but exact per-intent dollar amounts could shift slightly under a two-breakpoint model.
  • Discrepancy with stated metrics: measured cache_read_tokens / total_input_tokens = 71.8% (vs the 91% you cited) and token-weighted amortization = 4.38× (vs 5.89×). Closest match: cache-touched share of input = 88.2% (matches "91%" within rounding) — likely the same metric under a different name.
